# セントラルID 企画書

## 概要

セントラルIDは、複数のアプリケーションを横断して利用できる**共通ユーザー認証基盤**です。
「セントラルIDさえあれば、さまざまなアプリケーションが使える」をコンセプトに、認証・ユーザー管理・利用履歴の一元管理を提供します。

---

## 認証の基本方針

- **パスワードは保存しない**
- **メールアドレスは認証手段のひとつ**であり、必須項目ではない
- メール認証では**6桁のワンタイムコード**をメールに送り、コード入力でログインする（パスワードレス）
- ユーザーの主キーはセントラルIDが発行する**内部UUID**であり、メールアドレスや外部IDには依存しない
- 認証手段（メール・Google など）は複数紐付け可能

---

## 主要機能

### 1. 認証 API

外部アプリケーションに対して認証機能をAPIとして提供します。

- **メールアドレス認証**（パスワードレス・6桁ワンタイムコード）
- **ソーシャルログイン**（Google、GitHub など複数プロバイダー対応）
- **セッション管理**
  - アクセストークン発行・検証
  - リフレッシュトークンによるセッション延長
  - セッション切れ時の自動チェック
- **ログアウト**（トークン無効化）

### 2. ユーザー管理

- ユーザー登録・更新・削除
- ユーザーネーム変更
- **登録元アプリの記録**
  - 新規登録時にアプリIDを指定することで、どのアプリ経由で登録されたかを保持
- **ユーザー分類**（管理者がユーザーにタグ・グループを付与して分類）

### 3. 利用履歴の記録

セントラルIDを通じたすべての認証活動を記録します。

- いつログインしたか（タイムスタンプ）
- どこからログインしたか（IPアドレス、デバイス情報など）
- どのアプリを利用したか（アプリID）
- どの認証手段を使ったか（メール / Google など）

### 4. 管理者向け管理ページ

認証基盤の状況を可視化・操作するための管理UIを提供します。

- **ダッシュボード**：ユーザー数の推移、ログイン数の推移など
- **ユーザー一覧・検索・抽出**：条件を指定してユーザーデータをエクスポート
- **ユーザー分類**：グループ・タグの管理
- **利用履歴の閲覧**：ユーザーごとのログイン履歴・利用アプリ履歴

## 利用シナリオ

### 新規登録

#### A. メールアドレスで登録する場合

```
1. アプリ上で「メールアドレスで登録」を選択
2. メールアドレスを入力
3. セントラルIDが6桁のワンタイムコードをメール送信
4. ユーザーがコードを入力し、メールアドレスの所有を確認
5. セントラルIDが内部UUIDを発行してアカウントを作成
   - 登録元アプリIDを記録
6. 必要に応じてユーザーネームなどのプロフィールを設定
7. アクセストークン＋リフレッシュトークンを発行して登録完了
```

#### B. ソーシャル（Google など）で登録する場合

```
1. アプリ上で「Googleで登録」を選択
2. Google の OAuth 認証画面へリダイレクト
3. ユーザーが Google アカウントで承認
4. セントラルIDが Google のプロバイダーUID を受け取りアカウントを作成
   - 登録元アプリIDを記録
5. 必要に応じてユーザーネームなどのプロフィールを設定
6. アクセストークン＋リフレッシュトークンを発行して登録完了
```

> メールアドレスなしでも、ソーシャル認証のみで登録・利用が可能。

---

### ログイン

#### A. メールアドレスでログインする場合

```
1. アプリ上で「メールアドレスでログイン」を選択
2. メールアドレスを入力
3. セントラルIDが6桁のワンタイムコードをメール送信
4. ユーザーがコードを入力
5. コードが一致すれば認証成功
6. アクセストークン＋リフレッシュトークンを発行
7. ログイン履歴（日時・IP・アプリID・認証手段）を記録
```

#### B. ソーシャル（Google など）でログインする場合

```
1. アプリ上で「Googleでログイン」を選択
2. Google の OAuth 認証画面へリダイレクト
3. ユーザーが Google アカウントで承認
4. セントラルIDが Google のプロバイダーUID で既存アカウントを照合
5. 照合成功でアクセストークン＋リフレッシュトークンを発行
6. ログイン履歴を記録
```

---

### セッション維持・ログアウト

```
通常アクセス：
  アクセストークンをAPIリクエストに付与 → セントラルIDが検証して通過

アクセストークン期限切れ：
  リフレッシュトークンで自動更新 → 新しいアクセストークンを発行

リフレッシュトークンも期限切れ：
  再ログインを要求

ログアウト：
  アクセストークン＋リフレッシュトークンを無効化
  （他デバイスのセッションも全破棄するか選択式にするか要検討）
```

## データ設計（主要エンティティ）

| エンティティ | 主なカラム |
|---|---|
| ユーザー | UUID（内部ID）、ユーザーネーム、登録元アプリID、登録日時 |
| 認証プロバイダー | ユーザーUUID、プロバイダー種別（email / google 等）、プロバイダーUID（メールアドレスを含む） |
| ワンタイムコード | メールアドレス、コード（6桁）、有効期限、使用済みフラグ |
| トークン | ユーザーUUID、アクセストークン、リフレッシュトークン、有効期限 |
| 利用履歴 | ユーザーUUID、アプリID、認証手段、IPアドレス、デバイス情報、タイムスタンプ |
| アプリ | アプリID、アプリ名、登録日時 |
| ユーザー分類 | ユーザーUUID、グループ/タグ名 |

## 技術方針

- セントラルID専用のドメインで外部アプリと分離（例：`centralid.win`）
- バックエンド：Laravel（PHP）
- フロントエンド（管理ページ）：未定
- データベース：MySQL
- セッション・キャッシュ：ファイルストレージ

---

## 今後の検討事項

- 対応ソーシャルプロバイダーの優先順位（Google 優先か、GitHub・LINE なども同時対応か）
- アクセストークンの有効期限設計（短命トークン＋リフレッシュの具体的な期間）
- ワンタイムコードの有効期限・試行回数制限・リトライ間隔
- ログアウト時のセッション範囲（現在のデバイスのみ）
- メールアドレスを後から追加・変更する際のフロー
- レート制限・ブルートフォース対策
